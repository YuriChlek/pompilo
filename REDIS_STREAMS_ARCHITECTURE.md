# Redis Streams Architecture For Two Bots

## Мета

Організувати роботу двох ботів так, щоб:

- ринкові дані синхронізував лише один сервіс
- боти не дублювали синхронізацію між собою
- запуск ботів відбувався по факту готовності нових даних
- система залишалась стійкою до рестартів і тимчасової недоступності ботів

---

## Проблема поточного підходу

Якщо кожен бот самостійно:

- синхронізує свічки
- вирішує, коли дані вже готові
- стартує торговий цикл по власному scheduler

то виникають типові проблеми:

- дублювання навантаження на Binance і БД
- гонки між двома ботами
- ризик, що один бот бачить вже оновлені дані, а інший ще ні
- складніше гарантувати однаковий момент запуску після sync

---

## Базова ідея

Потрібно розділити систему на два типи компонентів:

1. `sync-service`
2. `bot-consumers`

`sync-service`:

- синхронізує дані з біржі для таймфреймів `1h`, `4h`, `1d`
- записує їх у PostgreSQL
- після успішного завершення запису публікує подію в Redis Stream

`bot-consumers`:

- не синхронізують дані самі
- отримують подію з Redis
- читають уже готові дані з PostgreSQL
- запускають власний analysis / execution цикл

---

## Чому Redis Streams, а не Pub/Sub

Для цього сценарію `Redis Pub/Sub` не підходить як основний механізм оркестрації:

- повідомлення не зберігаються
- якщо бот був недоступний у момент публікації, подія губиться
- немає нормального механізму `ack`

`Redis Streams` підходить краще, бо дає:

- збереження подій
- `consumer groups`
- підтвердження обробки через `XACK`
- pending messages
- повернення завислих повідомлень іншому consumer через `XAUTOCLAIM`

---

## Цільова схема

### Компоненти

- `sync-service`
- `PostgreSQL`
- `Redis`
- `bot-a`
- `bot-b`

### Потік даних

1. `sync-service` забирає нові свічки `1h`, `4h`, `1d` з біржі
2. `sync-service` записує свічки в PostgreSQL
3. після успішного commit `sync-service` робить `XADD` у Redis Stream
4. `bot-a` читає подію зі своєї `consumer group`
5. `bot-b` читає ту саму подію зі своєї `consumer group`
6. кожен бот читає потрібні дані з PostgreSQL
7. після успішної обробки бот робить `XACK`

---

## Stream і consumer groups

### Stream

Рекомендована назва stream:

```text
market-data-ready
```

### Consumer groups

Якщо обидва боти мають обробляти кожну подію, то вони не повинні сидіти в одній групі.

Правильна схема:

- group `bot-a`
- group `bot-b`

Це означає:

- кожна подія один раз доставляється в `bot-a`
- кожна подія один раз доставляється в `bot-b`

Якщо всередині одного бота буде кілька реплік, тоді:

- усі репліки `bot-a` працюють у group `bot-a`
- усі репліки `bot-b` працюють у group `bot-b`

У такому випадку:

- між репліками одного сервісу навантаження ділиться
- між різними сервісами подія не втрачається

## Формат події

Подія має описувати не самі дані, а факт, що конкретний набір даних уже готовий у БД.

Рекомендований payload:

```json
{
  "event_type": "market_data_ready",
  "symbol": "ETHUSDT",
  "timeframe": "1d",
  "close_time": "2026-05-01T00:00:00Z",
  "sync_run_id": "01JTXYZ...",
  "source": "binance",
  "version": 1
}
```

### Мінімально необхідні поля

- `event_type`
- `symbol`
- `timeframe`
- `close_time`
- `sync_run_id`

### Для чого потрібен `sync_run_id`

Він дозволяє:

- зв'язати подію з конкретним запуском sync
- простіше дебажити проблеми
- логувати, який саме sync породив запуск бота

---

## Коли публікувати подію

Подію треба публікувати лише після того, як дані вже надійно записані в БД.

Правильний порядок:

1. отримали дані
2. записали дані в PostgreSQL
3. commit завершився успішно
4. лише після цього зробили `XADD`

Інакше бот може отримати подію раніше, ніж дані реально стали доступними для читання.

---

## Рекомендована granularity подій

Є два практичні варіанти.

### Варіант 1. Одна подія на `symbol + timeframe + close_time`

Приклад:

- `ETHUSDT 1h 2026-05-01T12:00:00Z`
- `ETHUSDT 4h 2026-05-01T12:00:00Z`
- `BTCUSDT 1d 2026-05-01T00:00:00Z`

Плюси:

- простіша ідемпотентність
- простіше retry
- менший blast radius при помилці

Мінуси:

- більше подій

### Варіант 2. Одна подія на весь batch sync

Приклад:

- `1d batch ready for 7 symbols`

Плюси:

- менше подій

Мінуси:

- важче частково перевиконувати
- важче локалізувати проблему по конкретному символу

Для торгового бота практичніше `Варіант 1`.

---

## Правило ідемпотентності

Бот не повинен виконати один і той самий торговий цикл двічі для одного й того ж набору даних.

Ключ ідемпотентності:

```text
symbol + timeframe + close_time
```

Тобто навіть якщо:

- подія буде доставлена повторно
- consumer перезапуститься
- повідомлення буде reclaimed з pending

бот має перевірити, чи вже був оброблений цей ключ.

### Де зберігати стан обробки

Можливі варіанти:

- окрема таблиця в PostgreSQL
- окремий ledger / run log
- Redis key з TTL як допоміжний шар, але не як єдине джерело істини

Найнадійніше:

- стан обробки тримати в PostgreSQL

---

## Pending messages і recovery

Якщо бот прочитав повідомлення, але впав до `XACK`, повідомлення залишиться в pending.

Потрібно передбачити:

- читання нових повідомлень
- періодичний перегляд pending
- reclaim старих завислих повідомлень

Типове правило:

- якщо повідомлення в pending довше N хвилин, інша репліка цього ж бота може його забрати через `XAUTOCLAIM`

Це дозволяє переживати:

- crash consumer-а
- restart контейнера
- зависання воркера

---

## Блокування під час execution

Навіть при коректній обробці stream-подій бажано мати додатковий захист від паралельного запуску одного й того ж символу.

Рекомендація:

- використовувати per-symbol lock

Приклад ключа:

```text
lock:bot-a:ETHUSDT:d1
```

Такий lock потрібен, якщо:

- у бота кілька реплік
- analysis/execution може запускатися довше за очікуване
- існує ризик повторної доставки або overlap двох близьких подій

Lock не замінює ідемпотентність.
Він лише зменшує паралелізм по одному й тому ж об'єкту.

---

## Джерело істини

У цій схемі ролі мають бути чітко розділені:

- `PostgreSQL` — джерело істини для market data і стану обробки
- `Redis Stream` — механізм доставки сигналу “дані готові”
- `бот` — споживач події та виконавець торгового циклу

Бот не повинен приймати торгові рішення прямо з payload stream-події.
Payload має лише вказувати, що саме треба прочитати з БД.

---

## Outbox pattern

Для максимально надійної схеми в майбутньому варто розглянути `outbox pattern`.

Ідея:

1. `sync-service` в одній БД-транзакції:
   - пише market data
   - пише запис у `outbox_events`
2. окремий publisher-process читає `outbox_events`
3. publisher відправляє подію в Redis Stream
4. після успішної публікації позначає outbox event як published

Це вирішує проблему:

- дані записалися в БД, але процес впав до `XADD`

Для першої версії можна стартувати без full outbox, але для продового режиму це правильніша модель.

---

## Семантика доставки

Реалістична ціль для цієї системи:

- `at-least-once delivery`

Це означає:

- подія може прийти повторно
- тому consumer має бути ідемпотентним

Не варто проектувати систему так, ніби Redis Stream забезпечить “exactly once”.

## Рекомендований operational flow

### 1D

1. `sync-service` оновлює `1d` свічки
2. для кожного `symbol` робить `XADD market-data-ready`
3. `bot-a` отримує `1d`-подію
4. `bot-b` отримує `1d`-подію
5. кожен бот читає `1d` дані з PostgreSQL
6. кожен бот окремо запускає свою логіку
7. після успіху робить `XACK`

### 4H

1. `sync-service` оновлює `4h` свічки
2. для кожного `symbol` робить `XADD market-data-ready`
3. бот фільтрує події по `timeframe=h4`
4. виконується той самий цикл обробки

### 1H

1. `sync-service` оновлює `1h` свічки
2. для кожного `symbol` робить `XADD market-data-ready`
3. бот фільтрує події по `timeframe=1h`
4. виконується той самий цикл обробки

---

## Мінімальні технічні правила для майбутньої реалізації

- не використовувати Redis Pub/Sub як основний transport для цього сценарію
- використовувати один stream `market-data-ready`
- створити окрему `consumer group` для кожного типу бота
- payload події має містити `symbol`, `timeframe`, `close_time`, `sync_run_id`
- `timeframe` має підтримувати щонайменше `1h`, `4h`, `1d`
- публікувати подію лише після успішного запису даних у PostgreSQL
- реалізувати ідемпотентність по `symbol + timeframe + close_time`
- передбачити reclaim pending messages
- за потреби додати per-symbol lock
- PostgreSQL має залишатись source of truth

---

## Висновок

Для сценарію з двома ботами правильна модель така:

- один сервіс синхронізує дані
- Redis Streams повідомляє, що дані готові
- кожен бот має власну `consumer group`
- боти читають дані з PostgreSQL і працюють ідемпотентно

Це краще за підхід, де кожен бот самостійно синхронізує ринок або покладається на ненадійну `Pub/Sub`-доставку.
